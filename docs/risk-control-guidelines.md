# 风控场景指南

本文档面向需要登录高价值账号、对封号敏感的用户。

## 核心思路

**降低因网络环境不一致而触发平台风控的概率。**

不是「绕过风控」，而是「让你的网络行为看起来像一个正常用户」。

## 什么会触发风控

| 行为 | 风险 |
|------|------|
| 短时间内出口 IP 地理位置突变 | 高 |
| 同一会话内多个区域出口混用 | 高 |
| DNS 解析区域与出口区域不一致 | 中 |
| Web 和 API 请求走不同出口 | 中 |
| 频繁切换节点 | 中 |
| App 内和浏览器出口不同 | 中 |

## 操作建议

### 1. 使用严格模式

切换到 `configs/strict.conf`：
- 单一 DNS 源，减少解析不一致
- 精简规则集，减少意外命中
- 所有非 CN 流量统一走代理

### 2. 固定单一节点

- 不要使用「自动选择」「负载均衡」「回退」等策略组
- 选定一个节点后，在整个会话期间不要切换
- 如果必须换节点，建议间隔一段时间再登录

### 3. 将目标服务加入敏感列表

编辑 `rules/sensitive_services.list`，添加目标服务的所有相关域名：

```
DOMAIN-SUFFIX,target-service.com
DOMAIN-SUFFIX,api.target-service.com
DOMAIN-SUFFIX,auth.target-service.com
DOMAIN-SUFFIX,cdn.target-service.com
```

确保该服务的所有请求（登录页、API、CDN、认证）走同一出口。

### 4. 避免网络环境切换

在使用敏感服务期间：
- 不要在 Wi-Fi 和蜂窝之间切换
- 不要切换不同的 Wi-Fi 网络
- 不要开关代理

### 5. 保持 DNS 一致

- 使用 `dns/strict_dns.conf` 的单源配置
- 避免使用系统 DNS fallback
- 不要混用多个 DNS 协议（DoH / DoT / 传统 DNS）

## 敏感服务维护建议

`sensitive_services.list` 的维护原则：

1. **宁多勿少** — 同一服务的所有子域名都应加入
2. **优先级最高** — 该文件在规则链中位置靠前，优先命中
3. **定期检查** — 服务可能新增域名，关注是否有遗漏
4. **不要混用策略** — 同一服务的所有域名走同一个 PROXY 策略

## 本项目不能解决的问题

即使出口完全一致，平台仍可能基于以下维度判断：

- 设备指纹（浏览器 fingerprint、系统版本）
- Cookie 和本地存储
- 登录行为模式（时间、频率）
- 历史异常记录
- 账号本身的风险等级

**出口一致性是必要条件，不是充分条件。**
